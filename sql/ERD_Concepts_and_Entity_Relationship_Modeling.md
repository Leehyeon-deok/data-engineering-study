📘 ERD (Entity Relationship Diagram) 정리
1. ERD 개요

ERD란 DB를 구성하는 데이터의 논리적 구조를 시각적으로 표현한 모델

데이터를 엔터티(Entity) 로 정의하고
엔터티 간의 관계(Relationship) 를 표기법으로 표현

2. ERD 작성 순서 (⭐ 시험 단골)

엔터티 도출

엔터티 배치

엔터티 간 관계 설정

관계명 기술

관계의 차수 기술 (1:1, 1:N, N:M)

관계의 참여도 기술 (필수 / 선택)

3. 엔터티(Entity)
3.1 엔터티 정의

업무에서 관리해야 할 대상

식별자(PK) 를 반드시 가져야 함

두 개 이상의 인스턴스를 가짐

3.2 엔터티 분류
① 발생 시점 / 상속 관계 기준

기본 엔터티

독립적으로 생성 가능

주식별자를 스스로 가짐

중심 엔터티

업무의 핵심 역할

행위 엔터티

두 개 이상의 엔터티로부터 생성

트랜잭션 성격 (주문, 수강, 신청 등)

② 물리적 형태 존재 여부 기준

유형 엔터티

물리적 형태 존재 (사원, 상품)

개념 엔터티

물리적 형태 없음 (부서, 직무)

사건 엔터티

업무 수행 중 발생하는 이벤트 (주문, 결제)

4. 속성(Attribute)
4.1 속성 정의

엔터티의 구체적인 정보

더 이상 분리되지 않는 최소 데이터 단위

불필요한 속성은 최소화해야 함

4.2 속성값

속성은 집합

하나의 인스턴스는 하나의 속성값만 가짐

다중값 속성 발생 시 → 1차 정규화 대상

4.3 엔터티 · 인스턴스 · 속성 관계

하나의 엔터티 → 여러 인스턴스

하나의 인스턴스 → 여러 속성

하나의 속성 → 하나의 속성값

4.4 속성 분류
① 특성에 따른 분류

기본 속성: 업무에서 직접 도출

설계 속성: 시스템 구현을 위해 추가

파생 속성: 다른 속성으로부터 계산됨 (합계, 평균 등)

② 구성 방식에 따른 분류

PK (기본키): 인스턴스 유일 식별

FK (외래키): 다른 엔터티의 PK 참조

일반 속성

5. 도메인(Domain)

속성이 가질 수 있는 값의 범위

데이터 타입 + 길이 + 제약조건의 집합

6. 관계(Relationship)
6.1 관계의 종류

존재적 관계

엔터티 존재 자체로 연관

예: 사원 – 부서

행위적 관계

특정 행위로 발생

예: 고객 – 주문

※ ERD에서는 두 관계를 구분 없이 동일하게 표현

6.2 관계의 표현 요소

관계명: 관계의 의미

관계 차수

1:1

1:N

N:M

관계 선택성

필수 관계

선택 관계 (NULL 허용, 까치발 + ○)

7. 식별자(Identifier)
7.1 기본키(PK) 특징 (⭐ 시험 핵심)

유일성: 중복 불가

최소성: 최소 개수 속성

불변성: 변경되지 않음

존재성: NULL 불가

7.2 식별자 분류
① 대표성 기준

주식별자(PK)

보조식별자(AK)

유일성은 만족

참조 관계에는 사용 불가

👉 PK, AK 모두 후보키(Candidate Key)

② 생성 방식 기준

내부식별자

엔터티 내부에서 생성

외부식별자

다른 엔터티의 PK를 참조 (FK)

③ 속성 수 기준

단일식별자

복합식별자

④ 대체 여부 기준

본질식별자

업무상 본래 존재

인조식별자

인위적으로 생성 (대리키)

8. 식별자 관계 vs 비식별자 관계
① 식별자 관계 (실선)

부모 식별자가 자식의 주식별자

부모·자식 동일 생명주기

부모 삭제 시 자식도 삭제

② 비식별자 관계 (점선)

부모 식별자가 자식의 일반 속성

서로 다른 생명주기

🧠 시험용 한 줄 요약

ERD는 엔터티, 속성, 관계를 통해 데이터의 논리적 구조를 표현하며,
식별자와 관계의 차수·선택성이 핵심 설계 요소이다.
