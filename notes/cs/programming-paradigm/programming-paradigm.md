Programming Paradigms

프로그래밍 패러다임이란 프로그래머에게 프로그래밍 관점을 제공하는 개발 방법론이다.
문제를 어떻게 바라보고 어떻게 해결할 것인가에 대한 사고방식이다.

프로그래밍 패러다임은 크게 선언형과 명령형으로 나뉜다.
선언형 패러다임의 하위 개념으로 함수형 프로그래밍이 있으며,
명령형 패러다임은 객체지향 프로그래밍과 절차지향 프로그래밍으로 나뉜다.

선언형 프로그래밍과 함수형 프로그래밍

선언형 프로그래밍은 ‘무엇을’ 풀어내는가에 집중하는 패러다임이다.
프로그램의 실행 과정이 아니라 결과와 목적을 표현한다.

함수형 프로그래밍은 “프로그램은 함수로 이루어져 있다”라는 명제가 담긴 패러다임이다.
작은 순수 함수들을 조합하여 전체 로직을 구성한다.

순수 함수란 출력이 오직 입력에만 의존하는 함수로,
외부 상태를 변경하지 않고 부작용이 존재하지 않는다.

고차 함수란 함수를 매개변수로 받거나,
함수를 반환할 수 있는 함수를 의미한다.

고차 함수를 사용하기 위해서는 해당 언어가 일급 객체 특성을 가져야 한다.

일급 객체의 조건
1. 함수를 변수에 할당할 수 있다
2. 함수를 다른 함수의 매개변수로 전달할 수 있다
3. 함수가 함수를 반환할 수 있다

객체 지향 프로그래밍(OOP)

객체 지향 프로그래밍은 객체들의 집합으로 프로그램의 상호작용을 표현하는 방식이다.
데이터를 객체로 취급하고, 객체 내부에 선언된 메서드를 통해 로직을 수행한다.

설계에 많은 시간이 소요되며,
처리 속도는 다른 패러다임에 비해 상대적으로 느릴 수 있다.

객체 지향 프로그래밍의 특징
추상화: 복잡한 시스템에서 핵심 개념이나 기능만 추출하는 것
캡슐화: 데이터와 메서드를 하나로 묶고 외부에 일부를 은닉하는 것
상속성: 기존 클래스를 확장하여 재사용하는 것
다형성: 하나의 메서드나 클래스가 여러 방식으로 동작하는 것

오버로딩
같은 이름의 메서드를 여러 개 정의하는 것
매개변수의 타입이나 개수로 구분되며
컴파일 시점에 결정되는 정적 다형성이다.

오버라이딩
상위 클래스로부터 상속받은 메서드를 하위 클래스가 재정의하는 것
런타임 시점에 결정되는 동적 다형성이다.

객체 지향 설계 원칙 SOLID

SRP 단일 책임 원칙
모든 클래스는 하나의 책임만 가져야 한다.

OCP 개방 폐쇄 원칙
확장에는 열려 있고 수정에는 닫혀 있어야 한다.

LSP 리스코프 치환 원칙
부모 객체 자리에 자식 객체를 넣어도 프로그램이 정상 동작해야 한다.

ISP 인터페이스 분리 원칙
하나의 범용 인터페이스보다 여러 개의 구체적인 인터페이스가 낫다.

DIP 의존 역전 원칙
상위 모듈은 하위 모듈에 의존하지 않고 추상화에 의존해야 한다.

절차형 프로그래밍

절차형 프로그래밍은 로직이 수행되어야 할 순차적인 계산 과정으로 이루어진다.
코드의 흐름이 명확하고 가독성이 좋으며 실행 속도가 빠르다.

계산량이 많은 작업에 적합하며,
대표적으로 과학 연산, 머신러닝 배치 작업 등에 사용된다.

단점으로는 모듈화가 어렵고 유지보수성이 떨어진다는 점이 있다.

패러다임의 혼합

하나의 패러다임만 고집하기보다는
상황과 맥락에 따라 여러 패러다임을 조합하는 것이 바람직하다.

예를 들어
머신러닝 파이프라인은 절차 지향 패러다임,
거래 및 비즈니스 로직은 함수형 또는 객체 지향 패러다임을 적용하는 방식이 효과적이다.
